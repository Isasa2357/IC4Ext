# IC4Ext v1.0.1

## Added

- HW trigger / SW trigger configuration helpers
- `softwareTrigger()` API for D3D11 / D3D12 camera interfaces
- Typed acquisition lifecycle API:
  - `startAcquisition()`
  - `stopAcquisition()`
  - `isStreaming()`
  - `isAcquisitionActive()`
- Experimental `AcquisitionStartMode::Deferred`
- D3D12 `TimestampNearest` synchronization with selectable timestamp source
- `HostReceived` synchronization for independent camera clock domains
- D3D12 multi-camera synchronized display sample
- Optional recording of the same composed frame that is displayed
- OpenCV multi-camera analysis sample
- Motion-region, mean-luminance, changed-pixel-ratio, and region-count overlays
- IC4 device / transport diagnostic sample
- Public version constants in `IC4Ext/Version.hpp`
- Project version loaded from the root `VERSION` file

## Validated multi-camera startup

The validated production path uses normal stream setup followed by an acquisition pause before the next camera is prepared.

```text
camera 0: streamSetup(AcquisitionStart)
camera 0: AcquisitionStop command
camera 1: streamSetup(AcquisitionStart)
camera 1: AcquisitionStop command
...
start synchronization thread
start all camera worker threads
start all cameras through the typed startAcquisition() API
```

The public API remains typed:

```cpp
cameraThread.startAcquisition();
cameraThread.stopAcquisition();
```

For IC4Ext-owned D3D11/D3D12 captures, lifecycle control is not blocked by the worker thread's blocking read. External camera implementations remain serialized for compatibility.

`AcquisitionStartMode::Deferred` uses `ic4::StreamSetupOption::DeferAcquisitionStart`, but is experimental in v1.0.1. With the validated DFK 33UX252 / IC4 1.6.0.894 USB3Vision configuration, the second camera did not reliably produce frames through the deferred path. The production samples therefore use the validated prepare-stop lifecycle above.
